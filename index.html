Building an interface using React for hobbyist artists to showcase their work

Overview

In our digital age, we are in change of shaping who we are online. When it comes to entering the tech industry, this is even more important given the nature of this work. Futher, the value of a portfolio that showcases not only your technical experience but your personality and purpose is powerful. The objective of a customizable, digital mini gallery is to provide a platform that integrates the ability to showcase something more unique about an applicant as well as their development skills using React/JavaScript, HTML, and CSS. Further, the interactivity of the gallery through filtering, sorting, and favoriting allows for a more engaging experience for the user and increases the memorability of ones portfolio.

Competitive Comparison

Through conducting a competitive comparison on these three platform features, I found that while each one shares the same factors on a higher level, the implementation of such features are tuned for the underlying purposes of each company. Specifically, Amazon is a marketplace that mainly relies on you purchasing items incrementally, whereas Netflix and Spotify want you to continue your monthly subscription year over year.

üé∂ Spotify "Liked Songs"

Finding a middle ground, Spotify's "Liked Songs" feature allows one-click adding and removing, with extensive filtering options by genre and sorting by title, album, date added, and duration. This perfectly fits the use cases of quickly adding or removing a song as you are listening and of efficiently finding music to listen to. The genre filter options are hierarchically delineated, however only one genre can be applied at a time. Recommendations are added only when a user wants so that they do not clutter the interface.

Overall, I found that Netflix‚Äôs ‚ÄúMy List‚Äù is an oversimplified aggregator due to the lack of filtering and sorting functionality. Further, Amazon‚Äôs cluttered ‚ÄúLists‚Äù interface made a simple task overly complicated. On the other hand, Spotify‚Äôs cleverly filterable and sortable interface for ‚ÄúLiked Songs‚Äù struck the best balance for the purpose of the feature: for a user to view and browse all of their liked songs.

Feature List

For my interface, I am building a digital gallery for my origami creations, which I have made myself over the past few years. I hope for this gallery to grow over time, amassing all of my origami pieces, and for it to serve as part of my portfolio that not only showcases my React dev skills but also an artistic hobby that I love. Given my positive takeaways from Spotify‚Äôs ‚ÄúLiked Songs‚Äù interface and similar purpose to my origami gallery, I hope to draw the most inspiration from this interface and implement similar features, listed below.

  • Likes aggregator: users can interact with each origami piece through clicking on the like icon and aggregating a gallery of their favorites
  • One click adding and removing: allows users to like and unlike their favorite origami pieces using the same icon that changes appearance given selection state
  • Sorting: users can sort the origami gallery by title alphabetically, from A-Z
  • Filtering: users can filter the origami gallery by fold type and color; filters that are currently applied are marked accordingly in the dropdown to communicate the state of the interface to the user
  • Reset: users can reset the gallery to its original state, removing all applied sorts and filters

Design Process

Early Stage Sketches

When starting sketching, I started with the individual cards, as I knew the interface would build from those primary components. I drew out the "like" interaction, which was designed relative to the convention standardized by other social media platforms to invoke users' recognition. The interface is hierarchical, with the filters appearing first and then the gallery of origami pieces below. The likes aggregator gathers items in a column to create visual difference from the gallery in addition to line separator. Sketching out the full interface gave me a thorough mental model of what I wanted to accomplish in the low-fidelity wireframes.

Low-fidelity Wireframes

The low-fidelity wireframes were essential in the process of translating the design to a more uniform layout to make sure I was implementing my desired features without cluttering the interface. Then, I was able to move onto making all of my ideas come to life in the high-fidelity wireframes, accompanied by the style guide.

Style Guide

High-fidelity Mockups

The high-fidelity mockups fully implement the general layout of the low-fidelity wireframes, and all the components and attributes in the style guide. This final interface aims to amplify the art through neutral colors and a clean layout. One change that was made between the low-fidelity and high-fidelity versions was the coloring of the filter and sort buttons, which aim to delineate filtering and sorting to decrease the need for a user to recall which is which.
